# ai_worker

메인 백엔드(`scaffold/app/`)와 분리된 AI/RAG/멀티모달 추론 서비스입니다.
상세 설계는 RAG 담당(서현님)이 채워 넣기로 되어 있습니다 — 이 폴더는 그 뼈대만 잡아둔 것입니다
(`decision_log.md`의 "AI/RAG 워커 통신 방식" 미결사항 참고).

## 왜 두 가지 진입점이 있나요?

이 서비스는 성격이 다른 두 종류의 요청을 모두 처리해야 합니다.

- `sync_api.py`: **동기 응답**이 필요한 것. AI 챗봇 상담(T-LLM-2)의 실시간 RAG 검색처럼,
  스트리밍 응답 안에서 즉시 결과가 필요한 경우. 메인 백엔드가 이 작은 FastAPI 앱에 직접
  요청하고 바로 응답을 받는다.
- `queue_worker.py`: **비동기 큐 처리**가 되는 것. 알약/문서 인식(T-MED-1, T-DOC-1), 건강
  콘텐츠 생성(T-LLM-3)처럼 job을 등록해두고 나중에 폴링해도 되는 경우. Redis Streams를
  컨슈머로 소비한다.

## 아직 정해지지 않은 것 (RAG 설계 시 확정)

- 큐 종류/프로토콜 (Redis Streams 그대로 쓸지, 다른 걸 쓸지)
- `sync_api.py`의 정확한 엔드포인트 스펙 (요청/응답 스키마)
- 임베딩/비전 모델을 어떤 라이브러리로 로드할지, 상시 메모리에 띄워둘지
- ChromaDB(Server Mode)와의 연결은 이 서비스가 담당 (메인 백엔드가 아님)
