## RAG 개발 기간 동안의 화면별 대응 전략 (Tier 구조, 2026-07 추가)

서현님의 RAG Retriever/Context Binder가 완성되기 전까지, 이를 사용하는 화면은 아래 3단계로 나눠서 병행 개발한다. 다른 팀원들이 서현님의 RAG 완성을 기다리지 않고 각자 파트를 진행하기 위한 장치다.

| Tier | 대상 | RAG 필요 여부 | 처리 방식 |
| --- | --- | --- | --- |
| Tier 0 | T-TRCK-3(홈 위젯), T-ADH-1(순응도 히트맵), 3단계 정보등록 유도카드 | 없음 | RAG 완성 대기 없이 즉시 개발 |
| Tier 1 | T-LLM-3(건강 콘텐츠 생성 파이프라인) | 생성만 필요, 실시간 조회 아님 | APScheduler로 새벽 배치 생성 → MySQL `content_cache` 테이블 저장 → 화면은 캐시만 읽음 (TRD 기존 성공요건 "당일 캐시 재사용"과 일치) |
| Tier 2 | T-MED-2(약품 안전 정보 안내), T-DOC-2(인식 데이터 기반 맞춤 가이드) | 사용자별 실시간 판단 필요 | 응답 스키마를 먼저 확정하고, RAG 완성 전까지는 `recognition_service`(공통모듈)가 **규칙기반 stub**을 리턴. RAG 완성 후 Service 내부만 교체 (Router/프론트 무변경 — CODING_RULES.md의 계층 분리 원칙을 그대로 활용) |
| 실시간 필수 | T-LLM-2(AI 챗봇 상담) | 있음 | 서현님 RAG 파이프라인 완성 시점에 맞춰 통합, 우회 불가 |

> Tier 2가 Core(T-DOC-2)와 Phase 2(T-MED-2)에 걸쳐 있는 이유: 두 기능 다 "지금 당장은 스텁, 나중에 RAG로 교체"라는 같은 처리 방식을 쓰기 때문. Phase가 달라도 처리 전략은 동일하게 적용한다.

## RAG 아키텍처 결정 (2026-07 추가)

### AI/RAG/멀티모달 워커 분리 원칙
- RAG·이미지인식 등 무거운 추론은 메인 FastAPI 앱과 같은 프로세스에 두지 않고, **별도 서비스(가칭 `ai_worker`)로 분리**한다. 무거운 의존성(임베딩/비전 모델)이 메인 API 배포 이미지에 섞이는 것을 막고, 추론만 독립적으로 스케일할 수 있게 하기 위함(템플릿의 `ai_worker/` 분리 구조 참고 — 단, 템플릿 자체는 빈 껍데기라 내부 로직은 우리가 새로 설계함).
- 이 서비스는 성격이 다른 두 종류의 요청을 모두 처리해야 한다:
  - **비동기 큐 처리**: 알약/문서 인식(T-MED-1, T-DOC-1), 건강 콘텐츠 생성(T-LLM-3) — 이미 job 등록 후 폴링하는 구조라 그대로 맞는다.
  - **동기 응답**: AI 챗봇 상담(T-LLM-2)의 실시간 RAG 검색 — 스트리밍 응답 안에서 즉시 결과가 필요해 큐잉 방식이 안 맞는다. 별도의 빠른 동기 엔드포인트가 필요하다.
- 통신 방식(큐 종류, 동기 엔드포인트 프로토콜 등) 상세 설계는 RAG 담당(서현님) 설계 시점에 확정해 이 문서에 반영한다.

### Vector DB: ChromaDB Server Mode 채택 이유
- 임베디드(퍼시스턴트 클라이언트) 모드는 같은 프로세스 안에서만 데이터에 접근할 수 있어, APScheduler 배치 워커와 FastAPI 앱처럼 서로 다른 프로세스가 같은 컬렉션을 함께 읽고 쓰는 분산 워커 환경에서는 데이터를 공유할 수 없다.
- 이 문제를 피하기 위해 별도 ChromaDB 서버 프로세스 + HTTP 클라이언트 방식인 **Server Mode**를 채택한다 (위 "확정된 기술 결정" 표의 Vector DB 항목과 동일 결정).

### 청킹 전략: 필드 레벨 청킹
- 약품 정보를 하나의 긴 텍스트로 청킹하지 않고, **효능 / 용법 / 주의사항 / 상호작용 / 부작용 / 보관법** 등 필드 단위로 쪼개 각각을 별도 청크로 저장한다.
- 각 청크에는 원본 약품 ID·필드 종류 등 메타데이터를 함께 저장해, 검색 시 필요한 필드만 필터링해서 가져올 수 있게 한다.

### Collection 분리 원칙
- Collection은 목적별로 분리한다 (예: 약품정보, 건강뉴스, 대화이력 등을 서로 다른 Collection으로 관리).
- 서로 다른 개체 간 1-hop 관계(예: 약품 ↔ 부작용)는 별도 Collection이나 그래프 구조를 새로 만들지 않고, 메타데이터 필터로 대체한다.

### 대화 이력 관리
- 진행 중인 대화(현재 세션)는 Redis에 `thread_id` 기준으로 저장한다 (단기, 세션 성격 — 위 "확정된 기술 결정" 표의 "대화 컨텍스트" 항목 참고).
- 장기 보관·검색이 필요한 대화 맥락은 별도 `conversation_memory` Collection에 저장해 RAG 검색 대상에 포함시킨다.
