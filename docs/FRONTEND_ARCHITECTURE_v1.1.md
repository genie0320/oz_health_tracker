## 0. 기본 스택 (arch_00_overview / arch_01_client_api 기준, 변경 없음)

- React + Vite (SPA)
- React Router (클라이언트 사이드 라우팅, 서버는 `index.html` 1개만 서빙)
- Service Worker (Push 수신, 오프라인 캐싱, iOS는 `display:standalone` 필수)
- 인증: **JWT** (Access/Refresh Token, `Authorization: Bearer` 헤더 — 2026-07 결정, `decision_log.md` 참고, 기존 httpOnly 쿠키안에서 변경). Access Token은 메모리(React state)에만 두고 `localStorage`/`sessionStorage`에는 저장하지 않는 걸 권장 — XSS로 토큰이 털릴 범위를 최소화하기 위함. Refresh Token 보관 방식은 팀에서 별도 확정 필요 (메모리만 쓰면 새로고침마다 재로그인해야 하는 트레이드오프가 있음)

## 1. 계층 구조 — 백엔드의 "Router → Service → Repository"에 대응

백엔드가 계층을 나눈 이유(위에서 아래로만 호출, Router는 판단하지 않는다)를 프론트에도 그대로 적용합니다.

```
Page (화면)  →  Hook (상태 + 판단 로직)  →  api/ 함수 (fetch)  →  서버
```

| 레이어 | 여기서 해도 되는 것 | 여기서 절대 하면 안 되는 것 |
| --- | --- | --- |
| Page (`pages/`) | 컴포넌트 조립, 레이아웃, Hook 호출 | `fetch` 직접 호출 (CODING_RULES.md 기존 규칙), 복잡한 판단 로직 |
| Hook (`hooks/`) | 상태 관리, 조건 분기, 여러 api 함수 조합 | JSX 반환 (Hook은 로직만, 화면은 Page가 그림) |
| api (`api/`) | fetch 호출, 요청/응답 형태 변환 | 화면 관련 판단 ("로딩 중이면 스피너 보여줘" 같은 건 Hook/Page 몫) |

이렇게 나누면, 나중에 상태관리 방식이 바뀌어도(4번 참고) Page 코드는 거의 안 건드려도 됩니다.

## 2. 상태 관리

**베스트 프랙티스**: Redux, Zustand 같은 전역 상태 라이브러리 + React Query로 서버 상태 캐싱/동기화까지 체계적으로 관리.

**우리 팀 조정**: 전역 상태 라이브러리를 아예 도입하지 않습니다. 상태는 기본적으로 **탭(페이지) 안에서만 로컬로 관리**하고, 정말 여러 탭이 공유해야 하는 것(로그인한 유저 정보 정도)만 Context 1~2개로 최소화합니다.

**이유**: 상태관리 라이브러리는 selector, 미들웨어, 비동기 액션 패턴 등 개념이 하나 더 추가되는 건데, 초보 5인이 각자 다르게 쓰기 시작하면 "이 값은 어디서 바뀌는지" 추적이 오히려 더 어려워집니다. "탭 = 폴더, 그 안에서 알아서 관리"로 소유 경계를 명확히 하는 게, 다른 사람 상태에 실수로 손대는 사고를 원천 차단하는 데 더 효과적입니다.

## 3. 폴더 구조 (탭 단위 소유권)

```
frontend/src/
├── App.tsx                    # React Router 설정 (5탭)
├── pages/
│   ├── HomePage/
│   │   ├── HomePage.tsx
│   │   └── components/         # 이 페이지에서만 쓰는 컴포넌트
│   ├── TrackPage/               # '추적' — 탭 내부 분리 없이 섹션으로 구성
│   │   ├── TrackPage.tsx
│   │   └── components/
│   │       ├── AdherenceHeatmapSection.tsx   # T-ADH-1
│   │       ├── LifestyleSurveySection.tsx
│   │       ├── DietTrackingSection.tsx
│   │       └── HealthAppSyncSection.tsx
│   ├── ChatPage/                # '상담' — T-LLM-2
│   ├── InfoPage/
│   └── MorePage/                # '더보기' — 설정 등
├── components/
│   └── common/                  # 3개 이상 페이지에서 재사용될 때만 승격 (예: DisclaimerBanner)
├── api/                          # 엔드포인트당 함수 1개, api_spec_core_v1.yaml과 1:1
├── hooks/                        # 페이지 전용이 아닌, 여러 곳에서 쓰는 훅만
└── serviceWorker.ts
```

**베스트 프랙티스**: Atomic Design(atoms/molecules/organisms)처럼 컴포넌트를 역할별로 분류.

**우리 팀 조정**: 컴포넌트는 일단 그 컴포넌트를 쓰는 **페이지 폴더 안에** 둡니다. 정말 여러 페이지에서 반복해서 쓰는 게 확인되면 그때 `components/common/`으로 옮깁니다.

**이유**: Atomic Design은 "이게 atom이야 molecule이야"를 팀이 합의해야 하는데, 초보 팀에서는 이 판단 자체가 매번 논쟁거리가 됩니다. 페이지 폴더 안에 두면 "내 코드는 내 폴더 안에서만 수정한다"는 규칙이 자연스럽게 지켜지고, Git 충돌도 페이지 폴더 단위로 격리됩니다.

## 4. API 연동 레이어 & 타입

**베스트 프랙티스**: React Query/SWR로 서버 상태를 캐싱하고, 자동 재요청·무효화 전략을 체계적으로 관리. API 타입도 OpenAPI 스펙에서 자동 생성.

**우리 팀 조정**: 지금은 도입하지 않습니다. `api/` 폴더 함수 + 각 페이지 Hook에서 `loading / error / data` 3개 state를 다루는 동일한 패턴을 그대로 복사해서 씁니다. 타입도 `api_spec_core_v1.yaml`을 보고 `api/types.ts`에 **수동으로** 정의합니다.

**이유**: React Query는 캐시 무효화 전략이라는 개념이 하나 더 붙는데, 지금은 백엔드 API 자체가 Phase별로 계속 바뀌는 단계라 오히려 디버깅 지점만 늘어납니다. 자동 타입 생성 도구도 셋업/유지보수 비용이 있는데, 5인 규모에서는 "스키마 바뀌면 `api/types.ts`부터 고치고 팀에 공지"라는 수동 규칙(백엔드의 `api_spec_core_v1.yaml` 우선 원칙과 대칭)으로 충분합니다. 화면이 안정화되는 Phase 2~3 즈음 React Query 도입을 재검토하는 걸 추천합니다.

## 5. 스타일링

**베스트 프랙티스**: 별도 디자인 시스템 컴포넌트 라이브러리 + CSS-in-JS로 테마 관리.

**우리 팀 조정**: 유틸리티 클래스(Tailwind 등, 이미 정한 게 있으면 그걸로 통일) 하나로 몰아줍니다. 컴포넌트별 커스텀 CSS 파일을 새로 만들지 않습니다.

**이유**: 여러 명이 각자 `.css` 파일을 만들면 클래스명 충돌과 중복 스타일이 쌓입니다. 유틸리티 클래스는 스타일이 컴포넌트 파일 안에서 끝나기 때문에, 다른 사람의 CSS 파일을 열어볼 일 자체가 없어집니다.

## 7. Git 충돌 예방 (도구/규칙)

**베스트 프랙티스**: 자유로운 브랜치 전략 + 꼼꼼한 코드 리뷰로 충돌을 그때그때 해결.

**우리 팀 조정**:
- **Prettier + ESLint를 CI에 강제 적용** → 들여쓰기/세미콜론/따옴표 같은 "포맷 차이로만 생기는 무의미한 diff 충돌"을 원천 차단합니다.
- **import 순서 자동 정렬** (ESLint 플러그인) → 이것도 흔한 충돌 지점이라 자동화합니다.
- **PR은 한 페이지(탭) 폴더 위주로 작게 유지** → 여러 탭에 걸친 대형 PR 금지.
- 커밋 메시지는 백엔드와 동일하게 `[T-ID] 설명` 형식 유지.

**이유**: 초보 팀일수록 "코드 리뷰로 잘 걸러내자"보다 "애초에 충돌이 안 생기는 구조 + 자동화"가 훨씬 안전합니다. 사람이 스타일 차이를 매번 눈으로 잡아내는 건 실수하기 쉽습니다.

## 8. 요약 (한 줄씩)

| 항목 | 정석 | 우리 팀 조정 | 핵심 이유 |
| --- | --- | --- | --- |
| 상태관리 | Redux/Zustand + React Query | 탭 로컬 state + Context 최소 1~2개 | 개념 러닝커브, 소유 경계 명확화 |
| 컴포넌트 구조 | Atomic Design | 페이지 폴더 내 우선, 3+ 재사용 시 common 승격 | 분류 논쟁 방지, Git 충돌 격리 |
| API/타입 | React Query + 자동 타입생성 | 수동 fetch 패턴 + 수동 타입 정의 | API가 아직 자주 바뀌는 단계, 셋업비용 절감 |
| 스타일 | CSS-in-JS/디자인시스템 | 유틸리티 클래스 통일 | 클래스명 충돌 방지 |
| 충돌 방지 | 리뷰 중심 | Prettier/ESLint 강제 + 작은 PR | 자동화가 사람보다 안전 |
