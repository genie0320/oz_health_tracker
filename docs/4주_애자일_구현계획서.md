# 4주 구현 계획서 — Fail Fast 애자일 (5명, 비개발자 출신 + 바이브코딩)

> 전제: 5명 전원 4개월 전 코딩 시작(비전공), Claude/Antigravity 등 AI 페어코딩 활용, 자체구축 모델 1개 필수 포함
> 방법론: Week1=핵심 플로우 완성 → Week2=안정화 → Week3~4=고도화(이 단계에서 모델 반영). 매주 "동작하는 결과물"로 데모

---

## 0. 핵심 전략 — 모델은 뒤로, 플로우는 지금

**멘토 가이드**: 먼저 전체 플로우가 끊기지 않고 도는 핵심 기능부터 빠르게 완성한다(1주차, 다음주 금요일까지). 그다음 안정화·고도화 단계에서 세련되게 다듬는다(예: 룰베이스를 모델로 교체).

이 방향에 따라 계획을 다음과 같이 조정했습니다.

- **모델은 Week1 범위에서 완전히 제외**합니다. 자체구축 모델 후보(⑨ 백로그: REQ-MODEL-001~006)는 그대로 백로그에 둔 채, **Week1이 끝난 뒤 팀 논의로 "가장 쉬워보이는 것 & 영향도 있는 것"을 선택**해 Week2 이후에 착수합니다.
- **Week1은 알약 인식 대신 "복약안내서 촬영 → CLOVA OCR → 알림 데이터 생성"을 핵심 시나리오로 잡습니다.** 알약 이미지 분류보다 훨씬 다루기 쉬우면서도, 인증·업로드·외부API·DB·스케줄링까지 앱의 모든 레이어를 한 번씩 건드리는 얇은 수직 슬라이스입니다.
- 이 과정에서 만들게 될 "OCR 텍스트에서 복용정보를 뽑는 룰베이스 파서"는 그 자체로 **가장 자연스러운 모델화 후보**가 됩니다(멘토가 말한 "룰베이스를 모델로 바꾸는" 예시에 정확히 부합) — Week2 이후 모델 후보 논의 때 유력한 선택지로 5장에 정리해 두었습니다.

## 1. 주차별 로드맵

### Week 1 — 핵심 플로우 완성 (모델은 아직 잊는다)

**방향 전환**: 모델은 이번 주 범위에서 완전히 제외합니다. 자체모델은 ⑨ 백로그(REQ-MODEL-001~006 등)에 그대로 두고, **1주차가 끝난 뒤 팀 논의로 "가장 쉬워보이는 것 & 영향도 있는 것"을 골라 착수**합니다(2주차 이후).

**멘토 가이드 반영**: 이번 주는 "세련됨"보다 "덜컥거려도 끝까지 이어지는 것"이 목표입니다. 룰베이스/외부 API로 빠르게 전체 플로우를 완성하고, 이후 안정화·고도화 단계에서 다듬습니다(예: 룰베이스 파서 → 모델로 교체).

**핵심 시나리오(제안)**: 처방전의 **복약안내서를 촬영 → CLOVA OCR로 텍스트 추출 → 복용 시간·횟수 정보를 룰베이스로 파싱 → 복약 스케줄 생성 → 그 스케줄에 따라 알림 데이터가 실제로 생성되는지 확인**. 이 한 줄기가 이번 주의 "전체 플로우"입니다.

이 시나리오가 좋은 이유:
- `REQ-MED-001`(알약 사진 인식)보다 훨씬 다루기 쉽습니다. 알약은 "이 사진이 어떤 약인지 분류"해야 하지만, 복약안내서는 이미 텍스트로 인쇄돼 있어서 **OCR만 정확하면 나머지는 문자열 처리**로 풀립니다.
- 회원가입(최소)→업로드→인식→저장→알림 발생까지, 이 앱의 거의 모든 레이어(인증/파일업로드/외부API/DB/스케줄링)를 한 번씩 건드립니다. "화면이 넘어가고 API가 붙는지" 확인하기에 정확히 맞는 시나리오입니다.
- 지금 만드는 "룰베이스 텍스트 파서"가 나중에 모델로 교체하기 좋은 후보이기도 합니다(5장 참고).

#### Day별 계획 (월~금, 다음주 금요일 데드라인)

| Day | 목표 | 비고 |
|---|---|---|
| **Day 1 (월)** | 스택 확정, 리포지토리/CI 세팅, 최소 DB 스키마(users, prescriptions, schedules, notifications), **CLOVA OCR API 키 발급 후 사진 1장 넣어 텍스트가 나오는지 가장 먼저 확인** | 가장 위험한 외부 의존성(OCR)을 Day1에 검증 — 여기서 막히면 전체 일정에 영향 |
| **Day 2 (화)** | 화면 라우팅 뼈대(로그인 최소구현 → 업로드 화면 → 확인 화면 → 홈 화면, 데이터 없이 화면 전환만 확인), 백엔드 업로드 엔드포인트 + CLOVA OCR 연동 완료 | "화면이 넘어가는가"를 이 시점에 먼저 확인 |
| **Day 3 (수)** | OCR 결과 텍스트에서 "1일 3회 식후" 같은 복용정보를 뽑는 **룰베이스(정규식) 파서** 작성, 복약 스케줄 레코드 생성 | 실패해도 괜찮은 수준의 파서 — 정확도보다 "일단 뽑힌다"가 목표 |
| **Day 4 (목)** | 생성된 스케줄을 바탕으로 **알림 데이터가 DB에 실제로 쌓이는지** 확인(푸시 발송 인프라까지는 안 가도 됨 — "몇 시에 뭘 보내야 하는지" 레코드가 생기면 성공), 프론트-백엔드 연결 마무리 | REQ-NTFY-001의 최소 버전 |
| **Day 5 (금)** | **E2E 통합 테스트**: 사진 촬영 → OCR → 파싱 → 스케줄 → 알림데이터 생성까지 끊기지 않고 도는지 확인. 데모. 회고 + 2주차 킥오프(모델 후보 논의 준비) | 이번 주 목표 달성 기준 |

**이번 주 데모(완료 기준)**: 복약안내서 사진을 올리면, OCR로 글자가 뽑히고, 그 안의 시간정보로 알림 데이터가 생성된 것을 화면 또는 DB에서 확인할 수 있다. (로그인은 최소 수준이어도 무방, 이메일 인증 등 세부 검증은 생략 가능)

#### 역할 배분 (모델 담당 없이 5명 재배치)

| 인원 | 담당 | Week1 작업 |
|---|---|---|
| 2명 | OCR/파싱 페어 | CLOVA OCR 연동, 룰베이스 파서, 스케줄/알림 데이터 생성 로직 |
| 2명 | 화면 페어 | 로그인(최소), 업로드/확인 화면, 홈 화면, 화면 전환 |
| 1명 | 통합 담당 | 최소 인증·DB 스키마, 그리고 **매일 E2E가 끊기지 않는지 확인하는 역할**(멘토가 강조한 "전체 플로우"의 파수꾼) |

**Fail Fast 체크포인트 (금요일)**
- CLOVA OCR 인식률이 너무 낮으면 → 촬영 가이드라인 UI 추가 또는 테스트용 이미지 품질 통제로 우선 대응, 파싱 로직 탓인지 OCR 탓인지 반드시 구분해서 원인 확정
- 화면 전환이나 API 연결이 Day2까지 안 끝나면 → 스택을 더 단순한 조합으로 즉시 교체 검토
- Day5까지 E2E가 안 이어지면 → 2주차 신규 기능 착수를 미루고 이 루프 완성에 전원 재투입

*(관련 REQ: REQ-DOC-001/002(문서 OCR), REQ-NTFY-001(알림 발송) — 단, 식약처 API 대조나 진단명 추출 등은 이번 주 범위 아님)*

## 2. 인원 배분 원칙

5명 전원이 비슷한 시기에 코딩을 시작했다면, **완전한 FE/BE 분업보다는 페어(2인1조)**를 권합니다. 모델 담당은 Week1엔 존재하지 않고, **Week2 팀 회의에서 후보가 정해진 뒤 그때 배정**합니다(계속 갖고 갈 1명 또는 페어 하나를 지정 — ML은 실험→대기(학습)→재실험 반복이라 컨텍스트 전환 비용이 커서, 중간에 사람이 바뀌지 않는 게 좋습니다).

| 인원 | Week1 역할 | Week2 이후 |
|---|---|---|
| 2명 | OCR/파싱 페어 | 그대로 이어가거나, 모델 선정 시 이 페어가 모델 담당으로 전환(파서를 직접 만들어봤으니 모델화 맥락을 가장 잘 앎) |
| 2명 | 화면 페어 | 화면 지속 확장(안정화 UI, 신규 기능 화면) |
| 1명 | 통합 담당(E2E 파수꾼) | 인증/스키마/배포 등 인프라 지속 담당 |

바이브코딩 환경에서는 "AI가 짜준 코드를 이해하고 검증할 수 있는 사람이 최소 2명"인 페어 구조가, 1명이 전담하는 것보다 훨씬 안전합니다.

## 3. 이 계획의 변화 과정

| | 최초 2주 계획 | 4주 전환(1차) | **4주 최종(이번 반영)** |
|---|---|---|---|
| 알약 인식 | 외부 API 오케스트레이션 (P0, 무거움) | 자체 모델로 대체 | **범위에서 제외**, 대신 복약안내서 OCR로 대체 |
| 모델 시점 | 반영 안 됨 | Week1부터 최우선 검증 | **Week2 팀 논의 후 선정, Week3부터 개발** |
| Week1 목표 | (해당 없음) | 모델 정확도 PoC | **전체 플로우가 안 끊기고 도는가**(멘토 가이드) |
| 완성 방식 | Week2 끝에 한 번에 통합 | 매주 점진적 확장 | 매주 점진적 확장 유지 |

## 4. 이번 주(Week1) 이후 팀에 공유할 것

- Week1 끝나면 "무엇이 덜컥거렸는지"를 반드시 기록해두세요(OCR 정확도, 파싱 실패 케이스 등). 이 기록이 5장의 모델 후보 논의에서 "영향도"를 판단하는 근거가 됩니다.

## 5. Week2 논의를 위한 모델 후보 랭킹 (참고용)

최종 선택은 팀 논의로 정하시되, 참고할 수 있도록 ⑨ 백로그 항목 + Week1 경험에서 새로 떠오른 후보를 "난이도"와 "영향도" 기준으로 정리했습니다.

| 후보 | 난이도 | 영향도 | 비고 |
|---|---|---|---|
| **🆕 복약안내문 텍스트 파싱 모델화**(Week1의 룰베이스 파서를 대체) | 낮음 | **높음** | Week1에서 이미 데이터(OCR 결과물)가 쌓이기 시작하고, 핵심 플로우에 바로 꽂히므로 영향도가 가장 큼. 정형화된 짧은 텍스트 분류/추출이라 난이도도 낮은 편 |
| REQ-MODEL-003(복약 패턴 이상치 탐지) | 낮음 | 중간 | 이미지가 아닌 정형데이터 기반이라 접근이 쉬움. 다만 데이터가 충분히 쌓여야 의미가 있어 4주 내엔 임팩트가 제한적일 수 있음 |
| REQ-MODEL-002(음식 이미지 분류) | 중간 | 중간 | 공개 데이터셋이 많아 학습 자체는 할만하나, 이번 4주 핵심 플로우(복약)와 직접 연결되지 않아 영향도는 중간 |
| REQ-MODEL-001(알약 이미지 분류) | 높음 | 중간 | 원 취지에는 가장 부합하지만, 지금 핵심 플로우에서 이미 빠진 상태라 재편입 비용이 있음 |
| REQ-MODEL-005(PII 자동 탐지 NER) | 중간 | 낮음~중간 | 보안/개인정보 관련 항목이라 평가기준상 의미는 있으나, 사용자에게 보이는 임팩트는 작음 |
| REQ-MODEL-006(부작용 위험도 예측) | 중간~높음 | 중간 | 데이터(약물조합-부작용 이력)가 충분치 않으면 모델이 사실상 규칙 기반과 다르지 않을 위험 |

**제 의견을 굳이 보태자면**, 🆕 "복약안내문 텍스트 파싱 모델화"가 가장 유력해 보입니다 — Week1에 이미 만든 것을 발전시키는 것이라 이해도가 가장 높고, 실제 사용자 플로우에 바로 영향을 주고, 데이터도 이미 모이기 시작합니다. 하지만 이건 어디까지나 참고용이고, Week1을 실제로 겪어봐야 어떤 부분이 가장 아쉬웠는지(=가장 개선 가치가 큰지) 판단이 설 것 같습니다.

## 6. 4주 범위 밖 항목 — 명시적 후순위 목록 (공식 백로그 아님, 주의 필요)

PRD의 공식 백로그(⑦채팅/커뮤니티, ⑨자체모델)는 아니지만, 이번 4주 계획에도 이름이 등장하지 않는 항목들입니다. **"제외하기로 결정한 것"이 아니라 "스코프를 얇게 자르면서 빠진 것"**이므로, 팀이 한 번은 보고 "이대로 미뤄도 괜찮다"는 합의를 해두시길 권합니다. 5주차 이후 착수 후보로 보시면 됩니다.

| 카테고리 | 항목 | 비고 |
|---|---|---|
| 회원/인증 | REQ-AUTH-002(소셜로그인), REQ-AUTH-005(서포트그룹 연결) | 이메일 가입만으로 4주 데모는 가능. 서포트그룹은 가족 알림 기능(NTFY-004)과 묶여있어 그것도 함께 밀림 |
| 복약·건강기록 | REQ-MED-002(약품 안전정보), REQ-DOC-003(원본문서 보관/폐기), REQ-GOAL-002(목표기반 가이드), REQ-GOAL-003(달성 리포트), REQ-ADH-002(동기부여 피드백), REQ-GUIDE-001(질병 통합 대시보드), REQ-INFO-001(약품 상세조회), REQ-INFO-002(안전성 챗봇 QA) | 이 8개는 "복약 등록·알림"이라는 핵심 루프 자체엔 없어도 되는 부가가치 기능들 |
| LLM/AI | REQ-LLM-003(콘텐츠 생성 파이프라인), REQ-LLM-004(카드뉴스 이미지 생성), REQ-LLM-005(관리자 프롬프트 관리), REQ-LLM-006(뉴스 수집·매칭) | 챗봇(LLM-002)은 재편입했지만 이 4개는 부가 콘텐츠 기능이라 이번엔 보류 |
| 추적·대시보드 | REQ-TRCK-001(외부 헬스앱 연동), REQ-TRCK-002(패턴분석 가이드), REQ-DIET-001(식단 입력), REQ-DIET-002(영양분석), REQ-INFO-003(뉴스피드 UI), REQ-ACC-001(접근성 TTS) | 복약 외 영역(식단/운동/뉴스)이라 이번 4주 핵심(복약)과 결이 다름 |
| 알림 | REQ-NTFY-003(미루기), REQ-NTFY-004(가족알림), REQ-NTFY-005(부작용안내), REQ-NTFY-006(콘텐츠알림) | NTFY-001/002(발송·설정)만으로도 데모 가능. 나머지는 알림의 "편의 기능"층 |
| 보안/운영 | NFR-SEC-001(관리자 접근차단), NFR-STAT-001(통계 익명화) | 관리자 페이지 자체가 이번 4주 범위 밖이라 위험은 낮지만, **관리자 기능을 조금이라도 만든다면 이 두 개는 그때 반드시 같이 들어가야 함** |
| 성능/품질 | NFR-QUAL-004(사용성 기준 — 3~5단계/액션 접근성, 디자인 일관성) | 정량 기준이라 정성적으로는 이미 얇은 플로우 특성상 어느 정도 충족될 가능성 있음, 별도 검증은 생략 |

**요약**: 27개 항목이 이번 4주 계획 밖에 있습니다. 대부분(19개)은 핵심 복약 루프의 "부가가치·편의 기능"이라 미뤄도 제품이 안 죽지만, **보안 2건(SEC-001, STAT-001)만은 예외**입니다 — 관리자 기능을 조금이라도 만들 계획이 있다면 그 즉시 같이 넣어야 하는 항목이라, 미리 팀에 공유해두시는 게 좋겠습니다.
